# 难点解决

## Spring AI 调用 DashScope 参数类型错误

### 问题现象

Agent 调用大模型时，后端出现如下报错：

```text
'temperature' must be Float
```

后续还可能出现类似问题：

```text
模型维度格式不正确
dimensions 参数格式不正确
```

这些错误会导致 Spring AI 调用 DashScope 失败，推荐 Agent 无法进入完整的 LLM 意图解析和排序链路。

### 影响范围

- 智能推荐 Agent 无法稳定完成一次完整推荐。
- LLM 意图解析节点可能失败。
- LLM 排序节点可能失败。
- 系统会进入 fallback 分支，表现为没有真正调用大模型完成推荐。

### 根因

项目全局 Jackson 配置启用了：

```yaml
spring:
  jackson:
    generator:
      WRITE_NUMBERS_AS_STRINGS: true
```

同时项目中的 `JacksonCustom` 也启用了：

```java
builder.featuresToEnable(JsonWriteFeature.WRITE_NUMBERS_AS_STRINGS.mappedFeature());
```

该配置原本用于前端 Long 类型精度兼容，避免 Java Long 在浏览器端丢失精度。

但是 Spring AI 调用 DashScope 时，如果出站 HTTP 请求复用了全局 ObjectMapper，就会把请求体中的数字字段序列化成字符串。

例如：

```json
{
  "temperature": "0.2",
  "dimensions": "1024"
}
```

DashScope 兼容 OpenAI 接口对字段类型校验较严格，要求这些字段必须是 JSON number：

```json
{
  "temperature": 0.2,
  "dimensions": 1024
}
```

因此服务端返回参数类型错误。

### 定位过程

1. 检查 `application.yml`，确认 `temperature: 0.2` 本身写法正确。
2. 检查 `SpringAiRecommendationLlmClient`，确认已改为依赖 Spring AI Auto-Configuration，不再手动设置 `model`、`temperature`、`maxTokens`。
3. 继续排查发现 Spring AI 底层使用 `RestClient` 发起 HTTP 请求。
4. 检查项目全局 Jackson 配置，发现启用了 `WRITE_NUMBERS_AS_STRINGS`。
5. 推断 Spring AI 出站请求被全局 ObjectMapper 影响，导致数字参数被序列化成字符串。
6. 通过单元测试验证独立 ObjectMapper 可以让 `temperature`、`dimensions` 保持 JSON number。

### 解决方案

新增 Spring AI 专用 HTTP 客户端配置：

```java
@Configuration
public class SpringAiHttpClientConfig {

    @Bean
    public ObjectMapper springAiHttpObjectMapper() {
        return JsonMapper.builder()
                .findAndAddModules()
                .disable(JsonWriteFeature.WRITE_NUMBERS_AS_STRINGS)
                .disable(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS)
                .build();
    }

    @Bean
    @Order(Ordered.LOWEST_PRECEDENCE)
    public RestClientCustomizer springAiNumberSafeRestClientCustomizer(ObjectMapper springAiHttpObjectMapper) {
        MappingJackson2HttpMessageConverter converter =
                new MappingJackson2HttpMessageConverter(springAiHttpObjectMapper);
        return builder -> builder.messageConverters(converters -> replaceJacksonConverter(converters, converter));
    }
}
```

核心思路：

- 保留全局 Jackson 的 `WRITE_NUMBERS_AS_STRINGS`，不影响业务接口和前端 Long 精度兼容。
- 给 Spring AI 出站 HTTP 请求单独配置 ObjectMapper。
- Spring AI 调 DashScope 时，数字字段按 JSON number 序列化。
- 业务接口返回给前端时，仍然可以保持原来的数字转字符串策略。

### 验证方式

新增单元测试：

```java
@Test
void springAiHttpObjectMapperKeepsNumbersAsNumbers() throws Exception {
    ObjectMapper objectMapper = new SpringAiHttpClientConfig().springAiHttpObjectMapper();

    String json = objectMapper.writeValueAsString(Map.of(
            "temperature", 0.2,
            "dimensions", 1024
    ));
    JsonNode root = objectMapper.readTree(json);

    assertThat(root.get("temperature").isNumber()).isTrue();
    assertThat(root.get("dimensions").isNumber()).isTrue();
}
```

执行验证：

```bash
mvn -pl localmind-core-service -am -DskipTests compile
mvn -pl localmind-core-service '-Dtest=SpringAiHttpClientConfigTest,RecommendationContextBuilderTest' test
```

结果：

```text
BUILD SUCCESS
Tests run: 2, Failures: 0, Errors: 0
```

### 方案取舍

未选择直接关闭全局 `WRITE_NUMBERS_AS_STRINGS`，原因是：

- 项目中大量 ID 使用 Long。
- 前端 JavaScript Number 存在精度上限。
- 直接关闭可能影响已有接口返回格式，带来兼容风险。

最终选择隔离 Spring AI 出站 HTTP 序列化，是更小影响范围的修复方式。

### 面试表达

可以这样说明：

```text
我在接入 Spring AI 调 DashScope 时遇到过一个参数类型问题。application.yml 中 temperature 配置的是 0.2，但 DashScope 仍然提示 temperature must be Float。

一开始我以为是 Spring AI 配置绑定类型问题，后来排查发现项目为了前端 Long 精度兼容，全局 Jackson 开启了 WRITE_NUMBERS_AS_STRINGS，导致 Spring AI 出站请求中的 temperature、dimensions 等数字字段被序列化成字符串。

因为不能直接关闭全局配置，否则会影响前端 ID 精度兼容，所以我给 Spring AI 的 RestClient 单独配置了一个 ObjectMapper，禁用数字转字符串，只作用于大模型出站请求。

这样既保证业务接口返回格式不变，又保证 DashScope 请求体里的数值字段符合服务端类型校验。
```
