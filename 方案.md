# 智能推荐导购 Agent 实现方案

## 目标

构建用户端智能推荐导购 Agent。用户通过自然语言描述消费需求，Agent 将自然语言解析为结构化条件 criteria，并基于 criteria 调用后端工具查询候选门店，最终在对话中返回排序后的前三个推荐门店链接与推荐理由。

当前链路只解析和使用以下指标：

- 店铺类型：必须提供；未提供时 Agent 追问。
- 人数：用户提到时解析，用于预算估算。
- 预算：用户提到时解析，用于过滤或排序。
- 距离：在意图解析阶段归一化为 3km、5km、10km 三档。（用户未提到时默认按照10km来检索，因为必须使用GeoSeach，它不只是对距离的过滤还是对类型的过滤，后者是必须的）
- 营业时间：用户提到时解析，用于判断当前或指定时间段是否适合推荐。
- 综合评分偏好：用户提到“评分高、口碑好、人气高”等表达时解析，用于排序。

当前链路不引入用户历史消费记录和商家优惠券信息。

## 技术选型

- 使用 Spring AI 接入大模型，负责 LLM 调用、Tool Calling、结构化输出和会话记忆。
- 使用 LangGraph4j 编排 Agent 工作流，负责状态流转、节点拆分、条件分支、异常降级和 Checkpoint。
- 使用 BeanOutputConverter 将 LLM 输出转换为结构化 Java Bean，避免自由文本解析。
- 使用 Redis 保存短期会话 criteria、限流计数、熔断状态和推荐上下文。
- 使用 MySQL 保存门店、店铺类型等事实数据。
- 使用 Redis GEO 支持附近门店检索，距离范围固定映射为 3km、5km、10km。

## 工作流

```text
用户自然语言输入
-> 输入安全与业务领域判断
-> 读取 session 短期记忆
-> LLM 解析/微调结构化 criteria
-> BeanOutputConverter 转换为 Criteria Bean
-> 合并上一轮 criteria
-> 校验店铺类型是否存在
-> 类型缺失时追问
-> 基于 typeId + radiusMeters 调用 GEOSEARCH 工具
-> 获取符合条件且按距离排序的 shopId 列表，最多 200 条
-> MySQL 按 shopId 批量召回门店详情
-> Java 根据预算、营业时间进行二次过滤
-> 计算综合热度分并补充到候选门店
-> 裁剪无用字段，只给 LLM 排序必要信息
-> LLM 在候选集内输出 shopId 和推荐原因
-> 后端校验 shopId 和事实字段
-> 返回前三个门店链接
-> 异常时 Rule fallback
```

## Agent 节点设计

- InputGuardNode：负责输入长度、安全性、业务领域判断。
- SessionMemoryNode：读取 Redis 中当前 session 的上一轮 criteria。
- IntentExtractNode：调用 LLM 解析自然语言，直接完成条件抽取、距离档位归一化和多轮微调意图识别。
- CriteriaMergeNode：将本轮解析结果与短期记忆中的 criteria 做 slot merge。
- RequiredFieldCheckNode：校验店铺类型是否存在；缺失时进入追问节点。
- ClarifyNode：针对缺失店铺类型生成追问。
- GeoSearchToolNode：基于 typeId、用户坐标、radiusMeters 调用 Redis GEOSEARCH 工具，返回按距离排序的 shopId，最多 200 条。
- ShopRecallNode：根据 GEOSEARCH 返回的 shopId 批量查询 MySQL 门店详情。
- CandidateFilterNode：在 Java 层根据预算和营业时间对候选门店做二次过滤。
- ScoreBuildNode：计算门店综合热度分，并补充到候选门店。
- LlmRankNode：只接收必要候选字段，由 LLM 输出 shopId 和推荐原因。
- VerifyNode：校验模型输出的 shopId、字段合法性和推荐理由。
- ResponseNode：生成最终对话回复和前三个门店链接。
- FallbackNode：在 LLM 超时、输出不合法、熔断开启或候选校验失败时执行规则推荐。

## 短期记忆与多轮微调

Agent 必须具备短期记忆能力。用户不是每一轮都重新表达完整需求，可能在第一次推荐后继续微调条件。

示例：

```text
第一轮：今晚两个人想找附近评分高、200以内的美食店
第二轮：你推荐的太贵了，推荐便宜一点的
第三轮：这个太远了，我们想去5km内的
```

处理方式：

```text
第一轮生成完整 criteria
-> Redis 保存 session criteria
-> 第二轮 LLM 识别为 budget 调整，只更新预算相关字段
-> 第三轮 LLM 识别为 distance 调整，只更新 distanceLevel 和 radiusMeters
-> 每轮微调后都重新进入 GEOSEARCH、召回、过滤、评分、排序链路
```

短期记忆 Redis Key：

```text
agent:recommendation:session:{sessionId}
```

短期记忆内容：

```json
{
  "typeId": 1,
  "typeName": "美食",
  "peopleCount": 2,
  "budgetMax": 200,
  "distanceLevel": "3km",
  "radiusMeters": 3000,
  "startTime": "18:00",
  "endTime": null,
  "sortBy": "compositeScore"
}
```

slot merge 规则：

```text
本轮明确修改的字段覆盖上一轮
本轮未提及的字段沿用上一轮
店铺类型仍然是必填字段
距离微调表达必须重新归一化为 3km / 5km / 10km
```

## 自然语言解析

自然语言解析由 LLM 完成，不使用关键词匹配作为主流程。LLM 负责识别用户自然语言中的消费意图，并输出固定结构的 criteria。距离归一化也在 IntentExtractNode 中完成，而不是单独后置一个 DistanceNormalizeNode。

使用 Spring AI 的 BeanOutputConverter 将模型输出转换为 Criteria Bean。

示例输入：

```text
周末想和女朋友找个离我近一点、别太贵、评价好点的地方吃饭
```

结构化结果：

```json
{
  "typeName": "美食",
  "peopleCount": 2,
  "budgetPreference": "low",
  "budgetMin": null,
  "budgetMax": null,
  "distanceLevel": "3km",
  "radiusMeters": 3000,
  "startTime": null,
  "endTime": null,
  "sortBy": "compositeScore",
  "needLocation": true,
  "missingFields": []
}
```

店铺类型必须存在。若用户只说“帮我找个附近不错的店”，LLM 输出：

```json
{
  "typeName": null,
  "typeId": null,
  "missingFields": ["type"]
}
```

Agent 返回追问：

```text
你想找哪类店铺？例如美食、KTV、丽人、美容、美发、按摩、酒吧等。
```


## 距离归一化规则

后端附近门店能力只支持 3km、5km、10km 三档距离，因此 LLM 在 IntentExtractNode 阶段必须将自然语言距离表达归一化到固定档位。

```text
“很近”“走路能到”“附近”“离我近一点” -> 3km
“周边”“不要太远”“几公里内” -> 5km
“远一点也可以”“范围大一点”“十公里内” -> 10km
未提到距离 -> 默认 5km
```

后端仍会做兜底校验：

```text
radiusMeters 只允许 3000、5000、10000
非法值直接纠正为 5000
```

## 候选门店过滤

候选过滤由 Java 后端完成，LLM 不直接决定过滤结果。

- 类型过滤：GEOSEARCH 阶段已经基于 `shop:geo:{typeId}` 完成。
- 距离过滤：GEOSEARCH 阶段已经基于 radiusMeters 完成。
- 详情召回：MySQL 只根据 GEOSEARCH 返回的 shopId 批量查询门店详情。
- 预算过滤：Java 根据门店 avgPrice 和人数估算总价。
- 营业时间过滤：Java 根据 openHours 判断门店是否覆盖用户指定时间段。

MySQL 召回 SQL：

```sql
SELECT
    id,
    name,
    type_id,
    area,
    address,
    images,
    avg_price,
    score,
    sold,
    comments,
    open_hours
FROM tb_shop
WHERE type_id = #{typeId}
  AND id IN (...)
ORDER BY FIELD(id, ...);
```

说明：

```text
type_id 是冗余校验，防止 Redis GEO 索引脏数据
id IN 走主键索引，候选最多 200 条，查询成本可控
ORDER BY FIELD 用于保持 GEOSEARCH 返回的距离顺序
预算和营业时间不写入 SQL 条件，避免 open_hours 字符串解析导致 SQL 复杂且难以走索引
```

预算估算：

```text
estimatedTotalPrice = avgPrice * peopleCount
```

若未提供人数：

```text
peopleCount = 1
```

Java 二次过滤伪代码：

```java
List<ShopCandidate> filterCandidates(List<ShopCandidate> candidates, RecommendationCriteria criteria) {
    return candidates.stream()
            .filter(shop -> matchBudget(shop, criteria))
            .filter(shop -> matchOpenHours(shop, criteria))
            .toList();
}

boolean matchBudget(ShopCandidate shop, RecommendationCriteria criteria) {
    if (criteria.getBudgetMin() == null && criteria.getBudgetMax() == null) {
        shop.setBudgetMatched(true);
        return true;
    }

    if (shop.getAvgPrice() == null) {
        shop.setBudgetMatched(false);
        return false;
    }

    int peopleCount = criteria.getPeopleCount() == null ? 1 : criteria.getPeopleCount();
    long estimatedTotalPrice = shop.getAvgPrice() * peopleCount;
    shop.setEstimatedTotalPrice(estimatedTotalPrice);

    if (criteria.getBudgetMin() != null && estimatedTotalPrice < criteria.getBudgetMin()) {
        shop.setBudgetMatched(false);
        return false;
    }
    if (criteria.getBudgetMax() != null && estimatedTotalPrice > criteria.getBudgetMax()) {
        shop.setBudgetMatched(false);
        return false;
    }

    shop.setBudgetMatched(true);
    return true;
}

boolean matchOpenHours(ShopCandidate shop, RecommendationCriteria criteria) {
    if (criteria.getStartTime() == null && criteria.getEndTime() == null) {
        return true;
    }
    if (shop.getOpenHours() == null || !shop.getOpenHours().contains("-")) {
        return false;
    }

    String[] parts = shop.getOpenHours().split("-");
    LocalTime openStart = LocalTime.parse(parts[0].trim());
    LocalTime openEnd = LocalTime.parse(parts[1].trim());
    LocalTime userStart = criteria.getStartTime() == null ? openStart : LocalTime.parse(criteria.getStartTime());
    LocalTime userEnd = criteria.getEndTime() == null ? userStart : LocalTime.parse(criteria.getEndTime());

    return !openStart.isAfter(userStart) && !openEnd.isBefore(userEnd);
}
```

## 综合评分设计

用户表达“评分高、口碑好、人气高”时，不直接使用商家原始 score 排序，而是综合考虑原始评分 score 和销量 sold。设计目标是：评分高且销量高的门店得分最高；只有评分高但销量少，或只有销量高但评分一般，都不能获得过高排名。

### 字段归一化

原始评分归一化：

```text
scoreNorm = score / 50
```

其中 score 采用项目中的整数评分，例如 47 表示 4.7 分。

销量归一化使用对数函数，避免头部门店 sold 过大导致碾压：

```text
soldNorm = log(1 + sold) / log(1 + maxSold)
```

### 综合热度公式

```text
compositeScore =
0.45 * scoreNorm
+ 0.55 * soldNorm
+ 0.20 * scoreNorm * soldNorm
```

说明：

- sold 权重略高于 score，体现“高销量说明被更多用户验证过”。
- `scoreNorm * soldNorm` 是协同项，只有评分和销量同时高时才明显加分。
- 如果只有评分高但销量低，协同项较低，最终分不会太高。
- 如果只有销量高但评分一般，scoreNorm 会限制协同项和总分。

最终排序时可加入距离与预算轻微修正：

```text
rankScore =
compositeScore * 0.70
+ distanceScore * 0.20
+ budgetScore * 0.10
```

其中：

```text
distanceScore = 1 - min(distanceMeters / radiusMeters, 1)
budgetScore = 预算匹配则为 1，不匹配则为 0
```

## LLM 排序输入裁剪

ScoreBuildNode 计算完综合热度后，不把完整门店对象传给 LLM，只传排序必要字段。

LLM 输入：

```json
{
  "userQuery": "今晚两个人想找附近评分高、200以内的美食店",
  "criteria": {
    "typeName": "美食",
    "peopleCount": 2,
    "budgetMax": 200,
    "distanceLevel": "3km",
    "sortBy": "compositeScore"
  },
  "candidateShops": [
    {
      "shopId": 101,
      "typeName": "美食",
      "compositeScore": 0.92,
      "avgPrice": 86
    }
  ]
}
```

LLM 负责：

- 在候选集内选择前三个 shopId。
- 输出每个 shopId 的推荐原因。
- 推荐原因只能基于用户 query、criteria 和候选字段。

LLM 不负责：

- 编造门店。
- 编造评分、销量、价格、地址、营业时间。
- 返回候选集外的 shopId。
- 直接生成最终门店链接。

## 幻觉控制

- 候选集约束：LLM 只能从后端提供的 candidateShops 中选择。
- shopId 白名单校验：LLM 返回的 shopId 必须存在于候选集。
- 事实字段回填：前端展示的店名、图片、评分、价格、距离、销量、营业时间和链接全部来自后端真实数据。
- 推荐理由校验：推荐理由只能引用 LLM 输入中存在的字段，不能新增事实。
- 类型必填校验：typeId 为空时不查询门店，必须先追问。
- 距离档位校验：radiusMeters 只允许 3000、5000、10000。(如果用户没有说明距离我们默认为5000)
- 结构化输出：BeanOutputConverter 约束输出为 Criteria Bean 和排序结果 Bean。
- 候选裁剪：GEOSEARCH 最多返回 200 个 shopId；MySQL 批量召回详情后，Java 二次过滤；传给 LLM 前再裁剪字段，降低上下文漂移概率。

## 异常处理与降级

- 店铺类型缺失：不查询门店，直接追问类型。
- 定位缺失：提示用户授权定位或改用不按距离排序的推荐。
- LLM 超时：进入 Rule fallback。
- LLM 输出无法转换 Bean：重试一次，失败后进入 Rule fallback。
- LLM 返回候选集外 shopId：过滤非法结果，结果为空时进入 Rule fallback。
- 距离解析异常：默认使用 5km。
- 候选为空：返回无结果提示，不进入 LLM 排序。
- 连续失败：开启 Redis 熔断，在熔断窗口内跳过 LLM。
- 高频请求：基于用户维度进行 Redis 限流。

## Rule fallback

LLM 异常时使用规则排序返回结果。

```text
rankScore =
compositeScore * 0.70
+ distanceScore * 0.20
+ budgetScore * 0.10
```

排序流程：

```text
GEOSEARCH 获取最多 200 个 shopId
-> MySQL 批量召回门店详情
-> Java 预算过滤
-> Java 营业时间过滤
-> 计算 compositeScore
-> 计算 rankScore
-> 返回前三个门店
```

## 踩坑与解决方案

### 店铺类型缺失导致推荐范围过大

问题：用户只说“推荐附近不错的店”，如果直接查询会导致候选范围过大，推荐结果不可控。

解决：店铺类型作为必填字段。LLM 未解析出 typeId 时，Agent 先追问类型，不进入 GEOSEARCH 和门店召回。

### 距离表达过于模糊

问题：用户会说“附近”“别太远”“远一点也行”，无法直接映射为精确米数。

解决：在 IntentExtractNode 中直接将距离统一归一化为 3km、5km、10km 三档，并由后端再次校验非法距离值。

### 多轮条件被当成新对话

问题：用户第二轮只说“太贵了，便宜一点”或“想去5km内的”，如果没有短期记忆，会被当成一次缺少类型的新请求。

解决：Redis 保存 session criteria，LLM 识别本轮是条件微调还是全新需求，并通过 CriteriaMergeNode 合并上一轮条件，然后重新执行后续节点。

### 只按评分排序不稳定

问题：部分门店评分高但销量少，样本不足；部分门店销量高但评分一般，用户体验不一定好。

解决：使用 score 和 sold 的综合热度公式，并加入协同项，让评分和销量同时高的门店获得更高排名。

### 给 LLM 的门店字段过多

问题：完整门店对象包含地址、图片、营业时间、销量原始值等大量字段，容易增加 token 成本，也可能让模型引用不该展示的细节。

解决：LLM 排序阶段只提供 userQuery、criteria、shopId、typeName、compositeScore、avgPrice 等必要字段。最终展示字段由后端真实门店详情回填。

### 输出结构不稳定

问题：模型可能输出 Markdown、解释文字或不完整结构。

解决：使用 BeanOutputConverter 将输出约束为 Java Bean，转换失败时重试一次，仍失败则进入 Rule fallback。

### 门店幻觉

问题：模型可能返回不存在的门店或错误的门店字段。

解决：后端先召回候选集，LLM 只能返回候选集中的 shopId，最终展示字段全部由后端真实数据回填。

## 后续增强

- 增加门店标签体系，用于更细粒度地支持“适合约会、适合亲子、安静、热闹”等语义。
- 引入向量检索，用于匹配用户模糊偏好和门店描述。
- 增加推荐曝光、点击、下单埋点，动态调整排序权重。
- 增加 A/B 测试，对比 LLM 排序和规则排序的点击率。
- 增加可配置排序权重，支持运营侧调整销量、评分、距离、预算的权重。
